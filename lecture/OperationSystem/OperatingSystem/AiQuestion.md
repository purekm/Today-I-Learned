1. 인터럽트와 시스템 콜의 관계
"학습 내용 중 '사용자 모드에서 커널 모드로 전환하기 위해 시스템 콜을 호출하며, 이때 트랩(Trap)이 발생한다'고 하셨습니다. 그렇다면 소프트웨어 인터럽트인 트랩(Trap)과 하드웨어 인터럽트의 결정적인 차이점은 무엇이며, 운영체제가 이 둘을 구분해서 처리해야 하는 이유는 무엇일까요?"

=> 하드웨어 인터럽트는 비동기적으로, 외부적인 요인에 의해 발생하며 소프트웨어 트랩은 프로그램 내부에서 시스템 콜을 호출했을 때 처럼 동기저으로 발생함. 하드웨어 인터럽트는 키보드 IO 처럼 빠르게 반응해줘야 하지만, 소프트웨어 트랩은 요청의 처리가 목적이기 때문에 contextswitching으로 안전하게 커널을 넘겨줘야 함.
###
2. DMA(Direct Memory Access)의 필요성
"DMA 구조에서 인터럽트는 '블록 단위'로 발생한다고 정리하셨습니다. 만약 DMA가 없는 시스템에서 대용량 데이터를 I/O 장치에서 메모리로 옮긴다면, CPU의 효율성(Efficiency) 측면에서 어떤 심각한 문제가 발생할까요? 클라우드 서버처럼 트래픽이 몰리는 환경이라 가정하고 답변해 주세요."

=> DMA를 사용하지 않으면, CPU에 데이터를 옮길 때 마다 인터럽트를 걸어 서버를 못 쓰게 될 수 있음. 지속해서 서비스해야하는 서비스가 있으면 부하로 인해 기능을 못할 수 있음.
###

3. 가상화 vs 컨테이너 (Cloud Focus)
"포트폴리오에서 **Lambda(서버리스)에서 EC2(가상 머신)**로 아키텍처를 변경한 경험이 있으십니다. 학습 내용 중 '컨테이너는 호스트 OS의 커널을 공유한다'고 하셨는데, 보안적인 측면에서 볼 때 커널을 공유하는 컨테이너 방식이 하이퍼바이저 기반의 가상화(VM)보다 취약할 수 있는 지점은 어디라고 생각하시나요?"

=> 호스트의 커널을 공유하다보니, 호스트 커널에 문제가 발생하면 모든 컨테이너에 문제가 생길 수 있음
###

4. Storage Hierarchy와 AWS 서비스 매칭

"학습하신 Storage Hierarchy 개념을 AWS 서비스에 대입해 본다면, Main Memory(DRAM), SSD/HDD(NVM), Archive Storage 계층에 각각 대응하는 AWS 서비스(예: ElastiCache, EBS, S3 Glacier 등)를 속도와 비용 관점에서 설명해 주실 수 있나요?"

=> Main Memory는 ElastiCache로 메모리처럼 낮은 지연 시간을 제공하지만 휘발성이라는 특징, 
SSD/HDD의 경우 EBS로 로컬 디스크 역할을 수행하며 비휘발성 데이터를 저장,
Archive Storage는 S3 Glacier로 장기간 보유하는 서비스에 대입해볼 수 있을 것 같음
속도와 비용은 반 비례하고, Storage Hierarchy를 따라갈 것 같음

5. 시스템 보호와 권한 관리

"운영체제의 보안 기능에서 '권한 상승(Privilege Escalation)'과 '사용자 ID 관리'를 언급하셨습니다. 이를 AWS 환경에 적용한다면, **IAM Role과 Policy를 통해 최소 권한 원칙(Least Privilege)**을 지키는 것이 왜 중요한지 운영체제의 보호 메커니즘 관점에서 설명해 보세요."

=> 최소권한을 줌으로써 하는 역할을 확실히 하고, 다른 행동을 하지 못하게 제한함으로써 보안을 높일 수 있음
###

6. 가용성과 분산 시스템

"분산 시스템은 고가용성을 제공한다고 하셨습니다. CKA를 보유하고 계신데, Kubernetes 클러스터에서 특정 노드가 고장 났을 때 분산 시스템의 원리가 어떻게 작용하여 서비스 연속성을 보장하는지 'Self-healing' 개념과 연결해 설명해 주실 수 있나요?"

=> 분산 시스템은 단일 장애 지점을 제거하여 고가용성을 유지하는 것이 특징. k8s에서 또한 특정 노드가 고장났을 경우에는 워크로드인 pod들은 살아있는 다른 node로 self-healing을 통해 다시 실행하게 됨.
